# 项目开发指导规范

> 适用于本项目的 AI-Native 协作开发流程。此文档优先于口头约定，用于统一设计、实现、测试与交付标准。

## 1. 核心原则

- Design 是唯一真实来源（SSOT）。
- 文档分层，按需披露，先高层后细节。
- Design / Code / Test 必须保持一致。
- AI 负责执行、验证、总结；人类负责需求、设计、判断、验收。
- 任何变更都要同步更新设计、实现与测试。

## 2. 标准流程

任何功能都按以下顺序推进：

`HLD -> SDD -> Test -> Code -> Review -> Release`

说明：
- `HLD`：先明确目标、范围、约束、技术选型。
- `SDD`：再把实现细节、接口、数据结构、验收标准写清楚。
- `Test`：先定义测试，再实现代码。
- `Code`：按设计实现，避免自由发挥。
- `Review`：检查设计、代码、测试是否一致。
- `Release`：发布前必须完成清单检查。

## 3. 文档层级

文档建议按以下层级维护：

1. 系统定义
2. 服务/模块描述
3. 概要设计（HLD）
4. 详细设计（SDD）

要求：
- 文档使用 Markdown。
- 标题层级清晰，便于 AI 解析。
- 只写当前阶段需要的细节，避免一次写满。

## 3.1 本项目站点结构

本项目是偏开发者路线的个人站，当前建议的一级结构如下：

- `Home`
- `About`
- `Projects`
- `Writing`
- `Contact`

各页面职责：

- `Home`：一句话自我定位、当前状态、主项目入口、最新笔记入口、联系 CTA。
- `About`：背景、技术方向、擅长内容、不做什么、当前关注点。
- `Projects`：项目展示页，优先放个人站本身，其次放练习项目、工具、小型开源或课程成果。
- `Writing`：技术笔记、学习记录、踩坑总结、方法整理。
- `Contact`：邮箱、GitHub、其他社交账号、合作方式。

补充原则：

- 第一版不强求项目数量，个人站本身可以作为第一个项目。
- 不设置实验室页，避免引入不必要的内容压力。
- 先把内容容器搭好，后续再逐步补充内容。

## 4. 版本与变更

- 设计文档是版本管理的起点。
- 设计一旦变更，相关代码和测试必须同步调整。
- 变更影响较大时，应升级设计版本并说明影响范围。

建议的版本判断：
- `MAJOR`：架构或核心流程有重大变化。
- `MINOR`：新增功能或模块级变化。
- `PATCH`：小修复、小优化，不影响整体架构。

## 5. 测试分级

测试按层级推进：

- `UT`：单元测试，验证函数/方法逻辑。
- `API`：接口契约测试，验证请求与响应。
- `SIT`：系统集成测试，验证跨模块流程。
- `UAT`：用户验收测试，验证真实业务场景。

建议门槛：
- `UT` 覆盖关键逻辑。
- `API` 关键接口必须覆盖。
- `SIT` 覆盖核心业务流程。
- `UAT` 覆盖用户主路径。

## 6. 角色分工

- `Human`：提出需求、确认设计、做最终验收。
- `Architect`：负责架构和技术选型。
- `PM`：负责拆解任务、排期、验收标准。
- `Dev`：负责实现与单测。
- `QA`：负责测试、回归和质量门禁。
- `DevOps`：负责发布、环境和回滚策略。

## 7. 开发要求

- 先对齐意图，再开始实现。
- 先改设计，再改代码，再补测试。
- 代码提交前要能说明“改了什么、为什么改、怎么验证”。
- 不允许只改实现，不更新测试与文档。
- 不清楚的需求必须先澄清，不能靠猜。

## 8. 交付要求

每次交付至少包含：

- 变更说明
- 影响范围
- 验证结果
- 风险点
- 回滚思路（如适用）

## 9. 最小执行规则

任何新功能最少按这个节奏走：

1. 先写目标和边界
2. 输出 HLD
3. 输出 SDD
4. 定义测试
5. 再写代码
6. 跑验证
7. 再评审与合并

## 10. 优先级

如果本规范与临时口头说明冲突，以本规范为准。  
如果本规范与更高层项目设计文档冲突，以更高层设计文档为准。
